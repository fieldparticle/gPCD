For GLSL, the orientation/proximity/depth vocabulary is mostly already wired into the generated path.
Already updated:
ForceDynamics.glsl now has:ParticleProximityMagnitude(...)
ParticlePenetrationDepth(...)

ProcessParticleCollision(...) now fills ContactForceInput.penetrationDepth using ParticlePenetrationDepth(...).
ProcessWallCollision(...) uses the same helper with equal-radius ghost geometry.
ProcessBoundaryParticleWallCollision(...) also uses the helper.
AccumulateContactForce(...) still selects:penetrationDepth when CONTACT_FORCE_MEASURE == depth
overlapArea when CONTACT_FORCE_MEASURE == area

What still needs attention for GLSL parity:
Porting.tex should mention the new helper functions
Add ParticleProximityMagnitude and ParticlePenetrationDepth to the generated-method/export discussion.

Compute wrappers may need no geometry changes
The *P.comp files do not need to know orientation/proximity directly. They just call:
ProcessParticleCollision(...)
ProcessWallCollision(...)
ProcessBoundaryParticleWallCollision(...)
The generated functions now own the depth calculation.

Internal momentum cleanup is not yet mirrored in GLSL
Python now clears recoverable internal momentum when a source has no active contacts. GLSL does not yet have the equivalent. If we want exact parity, the *P.comp wrapper needs a way to know whether the source had at least one valid contact this frame, then clear:
P[SourceID].parms.yzw = vec3(0.0);
P[SourceID].Data.z = 0.0;
when no valid contact occurred.

Compile all active compute targets
We compiled BoundaryParticleReservoirPeriodicP.comp, but should also compile:
BoundarySpheresAndCubeP.comp
SimpleReservoirPeriodicP.comp

So the main missing GLSL work is not the penetration-depth math. That part is in. The main remaining behavioral mismatch is the no-active-contact internal momentum clearing.
