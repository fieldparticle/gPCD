\section{Discussion}

\subsection{Graphics-Pipeline Occupancy Generation}

Traditional collision-detection frameworks treat rendering as a
consumer of simulation state. Particle positions generated by the
simulation are subsequently transferred to the graphics pipeline for
visualization. In contrast, the gPCD framework integrates occupancy
generation directly into the graphics pipeline, allowing occupancy
construction and rendering to proceed concurrently.

The significance of this organization is architectural rather than
algorithmic. The framework does not introduce a new spatial data
structure; instead, it introduces a new execution model in which
graphics-pipeline operations contribute directly to broad-phase
collision detection. The resulting architecture enables occupancy
generation to be performed using hardware resources that would
otherwise be dedicated exclusively to rendering.

An additional benefit of the gPCD organization is that synchronization
is largely confined to the occupancy-generation stage. Atomic
operations are required only when reserving slots within shared cell
lists. Subsequent collision discovery is particle-centric and proceeds
without the construction of a global collision-pair structure, allowing
the narrow phase to map naturally onto massively parallel GPU
architectures.

\subsection{Scalability of the gPCD Framework}

The performance analyses collectively indicate that the gPCD framework
maintains near-linear scaling over the evaluated benchmark range.
Throughput measurements, time-per-particle trends, log--log
regressions, and curvature analyses all exhibit behavior consistent
with the complexity analysis presented earlier.

The measured scaling exponents remain close to unity for graphics,
compute, and total execution times, while the corresponding curvature
bounds remain small. Together, these observations suggest that no
individual stage introduces substantial nonlinear growth as particle
count increases.

Although throughput eventually enters a throughput-limited operating
region, the observed behavior remains consistent with increasing
resource utilization rather than a breakdown of the underlying scaling
characteristics of the framework.


\subsection{Memory Locality Considerations}

The locality benchmark demonstrates that memory organization affects
performance independently of algorithmic complexity. Both sequential
and random occupancy layouts retain the same asymptotic complexity;
however, sequential ordering provides improved memory locality and
therefore reduces execution time.

The approximately parallel trends observed in
Figure~\ref{fig:locality} suggest that locality primarily influences
the constant factor associated with memory access rather than the
underlying scaling characteristics of the framework. Consequently,
memory locality appears to affect implementation efficiency rather than
the asymptotic behavior of the framework itself.

\subsection{Implications for Large-Scale Particle Simulation}

\input{DensityStudies}

Figure~\ref{fig:density_progression} illustrates representative
particle densities spanning several orders of magnitude in particle
count. At low densities, particle distributions visually resemble dust
or sparse aerosols. Increasing density produces smoke- and fog-like
appearances before transitioning toward dense gaseous, liquid, and
solid-like representations.

These visualizations provide intuition regarding the practical scale of
particle systems that can be represented using the framework. The particle 
counts demonstrated in this work suggest that a single CPU--GPU system is 
capable of supporting particle-system scales that
have traditionally been associated with high-performance computing
(HPC) resources.
