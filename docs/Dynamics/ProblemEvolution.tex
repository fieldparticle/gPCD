\section{Problem Evolution}
\label{sec:problem-evolution}

The intent of this section is to follow the path of dynamics evolution.
Understanding that path makes the design issues and nomenclature visible.  It
also helps future development because the current model contains assumptions
about rules such as equal-radius particles, wall particles, and source-owned
contact state.  When those rules change, it should be clear where they came
from and why they existed.

\subsection{Observed Failure - Velocity Asymmetry After Collision in TwoParticleHorizontal}

\noindent\textbf{Commit hash:} \texttt{62ff53dbe45b11fb5aad8f5695d5f9513d2baf2a}

The first Geo collision response used a purely frame-local overlap impulse:

\begin{equation}
J = q A \Delta t.
\end{equation}

Here $q$ is the particle-owned pair stiffness, $A$ is the current overlap area,
and $\Delta t$ is the frame time step.  This rule was intentionally simple and
source-owned: each shader-shaped invocation updated only the current source
particle.

In \texttt{TwoParticleHorizontal}, total momentum remained conserved because
the two particles received equal and opposite velocity changes.  However, the
final speeds drifted slightly high:

\begin{equation}
v = 0.05 \quad \rightarrow \quad v \approx 0.050006.
\end{equation}

This is symmetric velocity drift.  It is invisible to total momentum:

\begin{equation}
P = \sum_i m_i v_i \approx 0,
\end{equation}

but visible in kinetic energy:

\begin{equation}
K = \sum_i \frac{1}{2}m_i \lVert v_i \rVert^2.
\end{equation}

The drift appeared at the end of rebound.  The particles were already
separating, but they were still slightly overlapped.  Because the old rule
continued applying $qA\Delta t$ until the overlap disappeared, the last small
overlap frames released a little extra velocity.

\subsection{Rejected Pairwise Fix}

A conventional pairwise elastic collision impulse can conserve both momentum
and kinetic energy:

\begin{equation}
J =
-\frac{(1+e)v_n}{\frac{1}{m_s}+\frac{1}{m_t}}.
\end{equation}

That approach updates both particles in one pair solve.  It is not allowed in
the current Geo path because the GLSL execution model is source-owned: one
invocation owns one source particle and must not write the target particle.

Therefore Python may not use a pairwise write-back shortcut.  The Python Geo
path must not become a separate reference model that fixes dynamics in a way
the shader cannot reproduce safely.

\subsection{Allowed Exception}

The accepted fix is source-owned internal normal momentum.  Each source-particle
contact slot may carry one persistent scalar:

\begin{equation}
I_{s,t} \ge 0.
\end{equation}

This value records the normal momentum removed from the source during
compression.  It is not a shared pair reservoir and it does not write the
target particle.

In the GLSL-shaped contact state, the value is stored in:

\begin{verbatim}
P[SourceID].contacts[slot].aux.z
\end{verbatim}

\subsection{Compression Rule}

The raw geometric impulse remains:

\begin{equation}
J_{raw} = q_{s,t} A_{s,t} \Delta t.
\end{equation}

During compression, the source can store only momentum that actually exists in
the source normal motion:

\begin{equation}
J_c =
\min\left(
J_{raw},
\max(0, m_s(v_s \cdot \hat{n}_{s,t}))
\right).
\end{equation}

The source velocity is updated by:

\begin{equation}
v_s' = v_s - \frac{J_c}{m_s}\hat{n}_{s,t}.
\end{equation}

The source-owned internal momentum ledger increases by the same amount:

\begin{equation}
I_{s,t}' = I_{s,t} + J_c.
\end{equation}

\subsection{Returning Rule}

When the source can no longer provide the full compression impulse, the contact
enters the returning phase.  During returning, the source may release only
stored internal momentum:

\begin{equation}
J_r = \min(J_{raw}, I_{s,t}).
\end{equation}

The velocity update uses the same source-owned write shape:

\begin{equation}
v_s' = v_s - \frac{J_r}{m_s}\hat{n}_{s,t}.
\end{equation}

The ledger is reduced:

\begin{equation}
I_{s,t}' = \max(0, I_{s,t} - J_r).
\end{equation}

This cap prevents the last tiny-overlap frame from releasing more momentum than
it stored during compression.

\subsection{Result}

For \texttt{TwoParticleHorizontal}, the internal momentum ledger drains to zero
at the end of contact and the kinetic energy returns to the starting value:

\begin{verbatim}
start_ke=0.002500000000000000
final_ke=0.002500000000000000
ke_drift=-8.67e-19
p1 vx=-0.050000000000000
p2 vx=+0.050000000000000
\end{verbatim}

The fix preserves the current GLSL architecture:

\begin{itemize}[leftmargin=*]
    \item each invocation still owns only one source particle;
    \item the target particle is read-only;
    \item the carried value is source-owned contact state;
    \item rebound cannot release more momentum than compression stored.
\end{itemize}

\subsection{Basic Symmetric Multi-Particle Functionality}

\noindent\textbf{Commit hash:} \texttt{01dbdcfbb1866a703727812b06606b50e83efedf}

After the two-particle internal momentum fix, the next failure was not a
single-pair rebound problem.  It was a source-capacity problem in symmetric
multi-particle contact.  In \texttt{ThreeParticle}, particle 1 can contact
particles 2 and 3 at the same time.  If each contact receives the full source
collision budget, the source spends the same momentum twice.

The first rule therefore had to be:

\begin{equation}
A_{s,total} = \sum_k A_{s,k}.
\end{equation}

Each contact receives an overlap-area weight:

\begin{equation}
w_{s,k} = \frac{A_{s,k}}{A_{s,total}}.
\end{equation}

\subsection{Rejected Multi-Contact Shortcuts}

Two shortcuts were rejected.  The first was averaging pair velocity
predictions.  That was an older practical method, but it is not a clean
geometric source-owned rule.  It hides the actual contact momentum accounting.

The second was applying the source area weighting only from the source side.
That fixed particle 1's split budget, but particles 2 and 3 each still saw a
one-contact context and could spend a full response against particle 1.  That
preserved the visual idea of weighting but still allowed momentum accounting to
be inconsistent.

\subsection{Accepted Symmetric Context Rule}

The accepted basic rule is still source-owned, but each contact is capped by
both sides' read-only contact context.  For a source contact $s,k$:

\begin{equation}
P_{s,k} = w_{s,k}P_{s,available}.
\end{equation}

The target context is computed the same way, but only read:

\begin{equation}
P_{t,k} = w_{t,k}P_{t,available}.
\end{equation}

The compression cap for the contact is:

\begin{equation}
P_{k,cap} = \min(P_{s,k}, P_{t,k}).
\end{equation}

The raw geometric impulse remains:

\begin{equation}
J_{raw,k} = q_{s,k} A_{s,k}\Delta t.
\end{equation}

The compression impulse is:

\begin{equation}
J_{c,k} = \min(J_{raw,k}, P_{k,cap}).
\end{equation}

Each contact slot still owns its own internal momentum ledger:

\begin{equation}
I'_{s,k} = I_{s,k} + J_{c,k}.
\end{equation}

During returning, the contact can release only its own stored momentum:

\begin{equation}
J_{r,k} = \min(J_{raw,k}, I_{s,k}).
\end{equation}

All active contact impulses are accumulated into one source-owned momentum
change:

\begin{equation}
\Delta \mathbf{p}_s =
-\sum_k (J_{c,k} + J_{r,k})\hat{n}_{s,k}.
\end{equation}

The source velocity is then updated once:

\begin{equation}
v_s' = v_s + \frac{\Delta \mathbf{p}_s}{m_s}.
\end{equation}

\subsection{Capture Rule}

Multi-contact debugging requires seeing the contact list directly.  The capture
system therefore records \texttt{contacts.csv} beside the existing
\texttt{momentum.csv} and per-particle CSV files.  The contact capture is
reporting-only.  It records the source, target, slot, overlap area, source and
target area weights, phase, stored internal momentum, applied impulse, normal,
center distance, and penetration depth.

The per-particle capture files also include \texttt{internal\_mom}, which is
the source particle's total stored internal momentum across its active contact
slots.

\subsection{Result}

For \texttt{ThreeParticle}, the symmetric multi-contact case now preserves
total momentum while allowing particle 1 to split its contact response across
particles 2 and 3:

\begin{verbatim}
ThreeParticle 240 frames:
momentum drift = -2.78e-17, 0
ke drift       = -0.0011817683521299304
max contacts   = 2
\end{verbatim}

This is basic symmetric multi-particle functionality.  It is not the final
multi-contact solver.  The current rule is still dissipative, and
\texttt{ThreeParticleChase} still exposes tunneling behavior.  Those are future
problem-evolution entries, not reasons to abandon the source-owned weighted
contact rule.

\subsection{Instantaneous Frame Snapshot and Sequential Frame Chain}

\noindent\textbf{Commit hash:} \texttt{674b92348cb2a72e4d5681d614b14f04551a4484}

After basic symmetric multi-particle contact was working, the next design issue
was not a new collision equation.  It was the meaning of the frame itself.  If
the model is instantaneous, then the collision response must be computed from
one frame-start state, not from values changed earlier in the same frame by
another source particle.

The problematic idea was ``collision-induced velocity changes.''  Treating
those changes as special inputs makes the model depend on intra-frame history:

\begin{verbatim}
source 1 changes velocity
source 2 reads that changed velocity
source 2 computes a different contact response
\end{verbatim}

That is a schedule artifact.  The same instantaneous state could have come from
many prior histories, so the response must not depend on which particle was
processed first.

\subsection{Accepted Snapshot Rule}

The accepted rule is to build a frame-start snapshot before contact processing
changes any particle state:

\begin{itemize}[leftmargin=*]
    \item \texttt{PosLocFrame} stores frame-start positions;
    \item \texttt{VelRadFrame} stores frame-start velocities.
\end{itemize}

Contact detection and contact geometry read from \texttt{PosLocFrame}.  Relative
normal velocity and available momentum read from \texttt{VelRadFrame}.  Collision
response still writes to the live source-owned particle state:

\begin{equation}
\text{read} = \text{frame-start snapshot},
\qquad
\text{write} = \text{live source state}.
\end{equation}

This preserves the instantaneous interpretation of the frame while still
allowing the frame to produce updated velocities and positions.

\subsection{Sequential Stage Rule}

The frame harness was also reshaped to make the program flow visible.  The
preferred style is a sequential, data-oriented stage chain with narrow
functions.  The main chain owns the decisions about which stages run:

\begin{verbatim}
GeoBeginFrame()
GeoApplyBeforeContactScanHook()
GeoResetFrameState()
GeoApplyStartRunHook()
GeoBuildFrameSnapshot()
GeoBuildParticleContactLists()
GeoBuildWallContactLists()
GeoResolveContacts()
GeoMoveParticles()
GeoEndFrame()
\end{verbatim}

This is easier to step through and debug because each stage has a clear kind of
activity.  It also keeps meaningful frame decisions visible.  For example,
\texttt{GeoBuildParticleContactLists()} and \texttt{GeoBuildWallContactLists()}
remain separate calls in the main chain instead of being hidden behind a broad
\texttt{GeoBuildContactLists()} umbrella.

\subsection{Result}

The Python Geo path now treats each frame as a snapshot-driven pipeline:

\begin{verbatim}
frame-start particles
-> frame snapshots
-> particle contact lists
-> contact resolution
-> moved particles
\end{verbatim}

The known test behavior remained stable:

\begin{verbatim}
TwoParticleHorizontal:
momentum drift = 0
ke drift       = -8.67e-19 after rebound

ThreeParticle 240 frames:
momentum drift = -2.78e-17, 0
ke drift       = -0.0011817683521299304

ThreeParticleChase:
still reaches GEO_ERROR_TUNNELING at frame 70
\end{verbatim}

This change is Python-side only.  GLSL export should be updated only on an
explicit export request, after the Python model is stable enough to checkpoint.
