\section{Problem Evolution}
\label{sec:problem-evolution}

The intent of this section is to follow the path of dynamics evolution.
Understanding that path makes the design issues and nomenclature visible.  It
also helps future development because the current model contains assumptions
about rules such as equal-radius particles, wall particles, and source-owned
contact state.  When those rules change, it should be clear where they came
from and why they existed.

\subsection{Observed Failure - Velocity Asymmetry After Collision in TwoParticleHorizontal}

\noindent\textbf{Commit hash:} \texttt{62ff53dbe45b11fb5aad8f5695d5f9513d2baf2a}

The first Geo collision response used a purely frame-local overlap impulse:

\begin{equation}
J = q A \Delta t.
\end{equation}

Here $q$ is the particle-owned pair stiffness, $A$ is the current overlap area,
and $\Delta t$ is the frame time step.  This rule was intentionally simple and
source-owned: each shader-shaped invocation updated only the current source
particle.

In \texttt{TwoParticleHorizontal}, total momentum remained conserved because
the two particles received equal and opposite velocity changes.  However, the
final speeds drifted slightly high:

\begin{equation}
v = 0.05 \quad \rightarrow \quad v \approx 0.050006.
\end{equation}

This is symmetric velocity drift.  It is invisible to total momentum:

\begin{equation}
P = \sum_i m_i v_i \approx 0,
\end{equation}

but visible in kinetic energy:

\begin{equation}
K = \sum_i \frac{1}{2}m_i \lVert v_i \rVert^2.
\end{equation}

The drift appeared at the end of rebound.  The particles were already
separating, but they were still slightly overlapped.  Because the old rule
continued applying $qA\Delta t$ until the overlap disappeared, the last small
overlap frames released a little extra velocity.

\subsection{Rejected Pairwise Fix}

A conventional pairwise elastic collision impulse can conserve both momentum
and kinetic energy:

\begin{equation}
J =
-\frac{(1+e)v_n}{\frac{1}{m_s}+\frac{1}{m_t}}.
\end{equation}

That approach updates both particles in one pair solve.  It is not allowed in
the current Geo path because the GLSL execution model is source-owned: one
invocation owns one source particle and must not write the target particle.

Therefore Python may not use a pairwise write-back shortcut.  The Python Geo
path must not become a separate reference model that fixes dynamics in a way
the shader cannot reproduce safely.

\subsection{Allowed Exception}

The accepted fix at this stage was source-owned internal normal momentum.
Initially this was carried in each source-particle contact slot:

\begin{equation}
I_{s,t} \ge 0.
\end{equation}

This value records the normal momentum removed from the source during
compression.  It is not a shared pair reservoir and it does not write the
target particle.

In that first implementation, the GLSL-shaped contact state stored the value in:

\begin{verbatim}
P[SourceID].contacts[slot].aux.z
\end{verbatim}

This was later superseded by the source-particle ledger described in the
``Source-Owned Internal Momentum Ledger'' entry.  The historical point remains
important: the value was source-owned, not a shared pair write-back.

\subsection{Compression Rule}

The raw geometric impulse remains:

\begin{equation}
J_{raw} = q_{s,t} A_{s,t} \Delta t.
\end{equation}

During compression, the source can store only momentum that actually exists in
the source normal motion:

\begin{equation}
J_c =
\min\left(
J_{raw},
\max(0, m_s(v_s \cdot \hat{n}_{s,t}))
\right).
\end{equation}

The source velocity is updated by:

\begin{equation}
v_s' = v_s - \frac{J_c}{m_s}\hat{n}_{s,t}.
\end{equation}

The source-owned internal momentum ledger increases by the same amount:

\begin{equation}
I_{s,t}' = I_{s,t} + J_c.
\end{equation}

\subsection{Returning Rule}

When the source can no longer provide the full compression impulse, the contact
enters the returning phase.  During returning, the source may release only
stored internal momentum:

\begin{equation}
J_r = \min(J_{raw}, I_{s,t}).
\end{equation}

The velocity update uses the same source-owned write shape:

\begin{equation}
v_s' = v_s - \frac{J_r}{m_s}\hat{n}_{s,t}.
\end{equation}

The ledger is reduced:

\begin{equation}
I_{s,t}' = \max(0, I_{s,t} - J_r).
\end{equation}

This cap prevents the last tiny-overlap frame from releasing more momentum than
it stored during compression.

\subsection{Result}

For \texttt{TwoParticleHorizontal}, the internal momentum ledger drains to zero
at the end of contact and the kinetic energy returns to the starting value:

\begin{verbatim}
start_ke=0.002500000000000000
final_ke=0.002500000000000000
ke_drift=-8.67e-19
p1 vx=-0.050000000000000
p2 vx=+0.050000000000000
\end{verbatim}

The fix preserves the current GLSL architecture:

\begin{itemize}[leftmargin=*]
    \item each invocation still owns only one source particle;
    \item the target particle is read-only;
    \item the carried value is source-owned contact state;
    \item rebound cannot release more momentum than compression stored.
\end{itemize}

\subsection{Basic Symmetric Multi-Particle Functionality}

\noindent\textbf{Commit hash:} \texttt{01dbdcfbb1866a703727812b06606b50e83efedf}

After the two-particle internal momentum fix, the next failure was not a
single-pair rebound problem.  It was a source-capacity problem in symmetric
multi-particle contact.  In \texttt{ThreeParticle}, particle 1 can contact
particles 2 and 3 at the same time.  If each contact receives the full source
collision budget, the source spends the same momentum twice.

The first rule therefore had to be:

\begin{equation}
A_{s,total} = \sum_k A_{s,k}.
\end{equation}

Each contact receives an overlap-area weight:

\begin{equation}
w_{s,k} = \frac{A_{s,k}}{A_{s,total}}.
\end{equation}

\subsection{Rejected Multi-Contact Shortcuts}

Two shortcuts were rejected.  The first was averaging pair velocity
predictions.  That was an older practical method, but it is not a clean
geometric source-owned rule.  It hides the actual contact momentum accounting.

The second was applying the source area weighting only from the source side.
That fixed particle 1's split budget, but particles 2 and 3 each still saw a
one-contact context and could spend a full response against particle 1.  That
preserved the visual idea of weighting but still allowed momentum accounting to
be inconsistent.

\subsection{Accepted Symmetric Context Rule}

The accepted basic rule is still source-owned, but each contact is capped by
both sides' read-only contact context.  For a source contact $s,k$:

\begin{equation}
P_{s,k} = w_{s,k}P_{s,available}.
\end{equation}

The target context is computed the same way, but only read:

\begin{equation}
P_{t,k} = w_{t,k}P_{t,available}.
\end{equation}

The compression cap for the contact is:

\begin{equation}
P_{k,cap} = \min(P_{s,k}, P_{t,k}).
\end{equation}

The raw geometric impulse remains:

\begin{equation}
J_{raw,k} = q_{s,k} A_{s,k}\Delta t.
\end{equation}

The compression impulse is:

\begin{equation}
J_{c,k} = \min(J_{raw,k}, P_{k,cap}).
\end{equation}

At this stage each contact slot still owned its own internal momentum ledger:

\begin{equation}
I'_{s,k} = I_{s,k} + J_{c,k}.
\end{equation}

During returning, the contact can release only its own stored momentum:

\begin{equation}
J_{r,k} = \min(J_{raw,k}, I_{s,k}).
\end{equation}

This per-slot ownership was later replaced.  First, internal momentum moved to
one source-total particle ledger.  Later, after the chase cases exposed that a
single source-total ledger was too coarse, the active Python model changed to
contact-owned ledgers stored under the source particle and keyed by target.
The contact slot remains a current-frame indexed record, not the owner of
persistent dynamics memory.

All active contact impulses are accumulated into one source-owned momentum
change:

\begin{equation}
\Delta \mathbf{p}_s =
-\sum_k (J_{c,k} + J_{r,k})\hat{n}_{s,k}.
\end{equation}

The source velocity is then updated once:

\begin{equation}
v_s' = v_s + \frac{\Delta \mathbf{p}_s}{m_s}.
\end{equation}

\subsection{Capture Rule}

Multi-contact debugging requires seeing the contact list directly.  The capture
system therefore records \texttt{contacts.csv} beside the existing
\texttt{momentum.csv} and per-particle CSV files.  The contact capture is
reporting-only.  It records the source, target, slot, overlap area, source and
target area weights, phase, stored internal momentum, applied impulse, normal,
center distance, and penetration depth.

The per-particle capture files also include \texttt{internal\_mom}, which is
the source particle's total stored internal momentum across its active contact
slots.

\subsection{Result}

For \texttt{ThreeParticle}, the symmetric multi-contact case now preserves
total momentum while allowing particle 1 to split its contact response across
particles 2 and 3:

\begin{verbatim}
ThreeParticle 240 frames:
momentum drift = -2.78e-17, 0
ke drift       = -0.0011817683521299304
max contacts   = 2
\end{verbatim}

This is basic symmetric multi-particle functionality.  It is not the final
multi-contact solver.  The current rule is still dissipative, and
\texttt{ThreeParticleChase} still exposes tunneling behavior.  Those are future
problem-evolution entries, not reasons to abandon the source-owned weighted
contact rule.

\subsection{Instantaneous Frame Snapshot and Sequential Frame Chain}

\noindent\textbf{Commit hash:} \texttt{674b92348cb2a72e4d5681d614b14f04551a4484}

After basic symmetric multi-particle contact was working, the next design issue
was not a new collision equation.  It was the meaning of the frame itself.  If
the model is instantaneous, then the collision response must be computed from
one frame-start state, not from values changed earlier in the same frame by
another source particle.

The problematic idea was ``collision-induced velocity changes.''  Treating
those changes as special inputs makes the model depend on intra-frame history:

\begin{verbatim}
source 1 changes velocity
source 2 reads that changed velocity
source 2 computes a different contact response
\end{verbatim}

That is a schedule artifact.  The same instantaneous state could have come from
many prior histories, so the response must not depend on which particle was
processed first.

\subsection{Accepted Snapshot Rule}

The accepted rule is to build a frame-start snapshot before contact processing
changes any particle state:

\begin{itemize}[leftmargin=*]
    \item \texttt{PosLocFrame} stores frame-start positions;
    \item \texttt{VelRadFrame} stores frame-start velocities.
\end{itemize}

Contact detection and contact geometry read from \texttt{PosLocFrame}.  Relative
normal velocity and available momentum read from \texttt{VelRadFrame}.  Collision
response still writes to the live source-owned particle state:

\begin{equation}
\text{read} = \text{frame-start snapshot},
\qquad
\text{write} = \text{live source state}.
\end{equation}

This preserves the instantaneous interpretation of the frame while still
allowing the frame to produce updated velocities and positions.

\subsection{Sequential Stage Rule}

The frame harness was also reshaped to make the program flow visible.  The
preferred style is a sequential, data-oriented stage chain with narrow
functions.  The main chain owns the decisions about which stages run:

\begin{verbatim}
GeoBeginFrame()
GeoApplyBeforeContactScanHook()
GeoResetFrameState()
GeoApplyStartRunHook()
GeoBuildFrameSnapshot()
GeoBuildParticleContactLists()
GeoBuildWallContactLists()
GeoResolveContacts()
GeoMoveParticles()
GeoEndFrame()
\end{verbatim}

This is easier to step through and debug because each stage has a clear kind of
activity.  It also keeps meaningful frame decisions visible.  For example,
\texttt{GeoBuildParticleContactLists()} and \texttt{GeoBuildWallContactLists()}
remain separate calls in the main chain instead of being hidden behind a broad
\texttt{GeoBuildContactLists()} umbrella.

\subsection{Result}

The Python Geo path now treats each frame as a snapshot-driven pipeline:

\begin{verbatim}
frame-start particles
-> frame snapshots
-> particle contact lists
-> contact resolution
-> moved particles
\end{verbatim}

The known test behavior remained stable:

\begin{verbatim}
TwoParticleHorizontal:
momentum drift = 0
ke drift       = -8.67e-19 after rebound

ThreeParticle 240 frames:
momentum drift = -2.78e-17, 0
ke drift       = -0.0011817683521299304

ThreeParticleChase:
still reaches GEO_ERROR_TUNNELING at frame 70
\end{verbatim}

This change is Python-side only.  GLSL export should be updated only on an
explicit export request, after the Python model is stable enough to checkpoint.

\subsection{Source-Owned Internal Momentum Ledger}

\noindent\textbf{Commit hash:} \texttt{486861f09a0118c1db62ea540f3848ec12e28bfc}

The basic symmetric multi-particle rule still carried internal momentum in
each contact slot.  That made contact slot identity too important.  Under the
instantaneous frame model, the current contact list should be rebuilt each
frame, and the next contact slot should simply be the next indexed slot:

\begin{verbatim}
slot = source.contactCount
source.contactCount += 1
\end{verbatim}

The persistent value is not the contact slot.  It is the source particle's
internal compressed state.  Therefore the accepted model moved internal
momentum to source-owned particle memory:

\begin{equation}
I_s \ge 0.
\end{equation}

In the Python model this is exposed as \texttt{particle.internal\_momentum} and
mirrored into \texttt{Data.z} for the GLSL-shaped particle layout.  The source
also carries a source-owned phase flag, \texttt{internal\_momentum\_phase}.

\subsection{Observed Failure - Shared Ledger Spent Sequentially}

After moving from per-slot ledgers to one source-owned ledger, the first
implementation spent and changed that ledger inside the contact loop.  This
created momentum drift in \texttt{ThreeParticle}.  At frame 105, particle 1 had
two contacts.  The first contact changed the source phase to returning before
the second contact was processed:

\begin{verbatim}
source 1, slot 0: compression + release
source 1, slot 1: release only
\end{verbatim}

The opposing source particles still saw one-contact compression contexts.  The
result was no longer symmetric, and total momentum drift appeared.

The fix was to make the source phase a frame-start read for every contact in
the source's current contact list.  The next source momentum and phase are
computed locally and committed once after all contacts are processed:

\begin{equation}
\text{read phase} = \phi_s^{frame-start},
\qquad
\text{write phase} = \phi_s^{next}.
\end{equation}

\subsection{Observed Failure - Tail Release Bias}

The frame-start phase fix removed the large drift, but a smaller drift appeared
near the end of returning.  At frame 120, particle 1 had a tiny internal
momentum balance left.  Slot 0 consumed the remaining ledger before slot 1
could release its share.  Because the normals were symmetric, spending the tail
on only one slot introduced a small net transverse impulse.

The accepted rule is that a source-owned total ledger must be shared across the
source's current contacts by overlap-area weight during release, just as source
compression capacity is shared during compression:

\begin{equation}
J_{r,k} =
\min\left(J_{raw,k}, w_{s,k}I_s + J_{c,k}, I_s\right).
\end{equation}

This keeps the source-owned memory model while preventing the indexed slot
order from deciding which contact receives the last remaining internal
momentum.

\subsection{Result}

After the source-owned ledger and area-weighted release fix:

\begin{verbatim}
TwoParticleHorizontal 200 frames:
momentum drift = 0

ThreeParticle 200 frames:
max momentum drift = 4.16e-17
final momentum     = (0.05, 0.0)
\end{verbatim}

This keeps the important architectural rules intact:

\begin{itemize}[leftmargin=*]
    \item contact slots are current-frame indexed records;
    \item at this stage, persistent internal momentum belongs to the source
    particle;
    \item target particles remain read-only during a source solve;
    \item source phase and frame snapshots are read consistently for the frame;
    \item source-owned release is shared across simultaneous contacts by area.
\end{itemize}

This source-total ledger rule was later superseded by contact-owned ledgers
keyed by target under the source particle.  The source-total rule remains in
the record because it explains why slot identity could not own persistent
memory and why stale contact history became visible in chase cases.

The next problem target is \texttt{ThreeParticleChase}, which should be treated
as a new problem-evolution step after this checkpoint.

\subsection{Frame-Local Closing Momentum for Chase Contacts}

\noindent\textbf{Commit hash:} \texttt{5eab35fe5bd4ec7d07824b7599306549c47d5ede}

The next failure was extreme overlap in chase contacts.  The simplest capture
was \texttt{TwoParticleChase}.  Particles 1 and 2 were closing, but the
source-normal availability test could report zero available momentum on one
side of the contact.  The result was a long sequence of valid closing contact
frames with no applied impulse:

\begin{verbatim}
rel_vn < 0
weighted_available_momentum = 0
applied_impulse = 0
\end{verbatim}

The contact response did not act until the particles were near the tunnel
threshold.  That prevented immediate pass-through, but it allowed an
unacceptably large overlap area.

\subsection{Rejected Interpretation - Persistent p\_zero}

The needed value resembles the old \texttt{p\_zero} idea, but the old
\texttt{p\_zero} carried collision-episode meaning: zero-depth calculations,
first-contact assumptions, and persistent state.  That does not fit the
current Geo conventions.

The accepted interpretation is strictly frame-local:

\begin{equation}
P_{close,k} =
\mu_{s,t_k}\max(0,-v_{n,k}),
\qquad
\mu_{s,t_k} =
\frac{m_s m_{t_k}}{m_s + m_{t_k}}.
\end{equation}

This value is computed from the frame-start velocity snapshot and the current
contact normal.  It is not stored as episode memory and it does not require a
pairwise write-back.

\subsection{Result}

Replacing the source-normal availability test with reduced-mass closing
momentum made \texttt{TwoParticleChase} conservative:

\begin{verbatim}
TwoParticleChase:
momentum drift      = numerical zero
closest p1-p2 range = 0.4660438566522238
radius sum          = 0.5
\end{verbatim}

\texttt{ThreeParticleChase} also has much better geometric behavior because
the chase contact no longer waits until an extreme overlap before responding.
However, it still has momentum drift in the multi-contact phase.  The next
problem target at that stage was therefore \texttt{ThreeParticleChase}
momentum drift.  The following entry records that fix and the kinetic-energy
problem it exposed.

\subsection{Frame-Local Pair Planning and Contact-Owned Ledgers}

\noindent\textbf{Commit hash:} \texttt{49b81b5162815932d9b05dcc69059ef1edc68034}

The next failure in \texttt{ThreeParticleChase} was total momentum drift during
multi-contact chase frames.  The underlying issue was that each directed source
side could still arrive at a slightly different compression or release amount.
Those differences were small, but once source-owned writes were applied they
were enough to move total momentum.

\subsubsection{Top-Level Fix}

The accepted fix was to separate planning from application:

\begin{verbatim}
Build frame snapshot
Build contact lists
Compute planned impulse per unordered pair from snapshot data
Apply planned impulses through source-owned writes
Move particles
\end{verbatim}

For an unordered pair $(a,b)$, each directed side computes a candidate impulse
from snapshot geometry, snapshot velocity, overlap-area weighting, and its own
contact ledger.  The pair then chooses one planned magnitude:

\begin{equation}
J_{plan,a,b}
=
\min\left(J_{candidate,a\rightarrow b},
          J_{candidate,b\rightarrow a}\right).
\end{equation}

Both directed source writes use that same planned magnitude.  This keeps the
GLSL-shaped ownership rule: the source invocation writes only the source
particle, but the impulse magnitude is pair-symmetric before either source is
updated.

\subsubsection{Ledger Change}

The source-total internal momentum ledger solved slot-order drift, but it was
too coarse for sequential or overlapping contacts.  A later contact could
inherit momentum stored by an earlier contact, and that made starting-overlap
and chase cases harder to reason about.

The current Python model therefore stores internal momentum under the source
particle, keyed by target particle:

\begin{verbatim}
particle.contact_internal_momentum[TargetID]
particle.contact_internal_phase[TargetID]
\end{verbatim}

The contact slot still does not own persistent memory.  The slot is rebuilt
each frame and reports the active target's ledger value.  When a target is no
longer in the source's current contact list, that target ledger is pruned.
The source particle's \texttt{Data.z} is now a total report value:

\begin{equation}
I_s = \sum_t I_{s,t}.
\end{equation}

\subsubsection{Current Result}

The planning change removed the measured momentum drift in the active
multi-particle tests:

\begin{verbatim}
ThreeParticle 150 frames:
max momentum drift = 6.94e-18
final KE drift     = -2.4771967621966314e-04

ThreeParticleChase 750-frame attempt:
max momentum drift = 4.16e-17
final KE drift     = -1.0677932798014234e-03
first error        = frame 464, GEO_ERROR_PARTICLE_OUT_OF_BOUNDS
\end{verbatim}

This shows that pair planning fixed the momentum symmetry problem, but it did
not solve kinetic-energy balance.  The stronger rebound also exposed the next
problem: wall or boundary behavior must be handled explicitly rather than being
hidden by damping.  The next work should start with
\texttt{ThreeParticleChase} kinetic-energy drift and the out-of-bounds/wall
case.

\subsection{Multi-Contact Kinetic-Energy Drift Diagnostics}

\noindent\textbf{Commit hash:} \texttt{f9e97e61c7550de3f9f3b2bd07c6ced8dd558d00}

After frame-local pair planning removed total momentum drift, the remaining
problem was kinetic-energy drift in the three-particle cases.  The important
observation was that total momentum could be correct while the contact impulse
exchange was still physically wrong:

\begin{equation}
\sum_i \Delta \mathbf{p}_i \approx 0,
\qquad
\sum_i \Delta K_i \ne 0.
\end{equation}

This reframed the problem.  The first question was no longer whether
\texttt{ThreeParticle} and \texttt{ThreeParticleChase} lost total momentum.
They did not.  The question became where the balanced but wrong momentum
exchange was introduced.

\subsubsection{Capture and Cleanup Foundation}

The capture path was tightened first.  New runs now clear existing
\texttt{*.csv} capture files from the configured \texttt{run\_debug\_dir}
before writing new rows.  The runner also prints the number of cleared capture
files and the directory path.  This prevents old capture rows from being
mistaken for current-run data, especially when a run is stopped manually before
the configured end frame.

The active batch comparison set is:

\begin{verbatim}
TwoParticleHorizontal
TwoParticleAngled
TwoParticleChase
ThreeParticle
ThreeParticleChase
\end{verbatim}

The two-particle cases remain useful baselines because they conserve both
momentum and kinetic energy under the current model.

\subsubsection{Source-Level Energy Diagnostics}

The first diagnostic layer added source-level impulse accounting to
\texttt{contacts.csv}.  For each source and frame, the capture records the net
source momentum change, the kinetic-energy change implied by that net impulse,
the sum of contact-by-contact kinetic-energy estimates, and the cross term
between the source net impulse and the per-contact estimates.

This showed that the final kinetic-energy drift equals the cumulative contact
response.  It was not introduced by position movement or reporting:

\begin{verbatim}
ThreeParticle:
actual final KE drift = -0.00024771967622

ThreeParticleChase:
actual final KE drift = -0.000400429449764
\end{verbatim}

Momentum drift stayed at floating-point noise.

\subsubsection{Shadow Planner Rule}

One proposed fix was to adjust source contact candidates by evaluating the
source net impulse before pair planning.  This was too risky to promote
directly.  The accepted workflow for this stage is a shadow planner: compute
alternative impulse plans beside the real dynamics, record their predicted
effect, and do not change particle motion.

The applied dynamics therefore remain:

\begin{verbatim}
directed candidates
unordered pair-compatible plan
source-owned velocity writes
\end{verbatim}

The shadow path is diagnostics only.

\subsubsection{Source-Scaled Shadow Planner}

The first shadow experiment kept the already-weighted contact candidates but
scaled a source's compression candidates as a set before pair compatibility was
computed.  The idea was to preserve the source-level net impulse intent better
than independent pair-local clipping.

The shadow result improved the predicted kinetic-energy loss but did not
eliminate it:

\begin{verbatim}
ThreeParticle:
actual KE estimate = -0.00024771967622
source shadow      = -0.000228391853378

ThreeParticleChase:
actual KE estimate = -0.000400429449764
source shadow      = -0.000350215634333
\end{verbatim}

This means source-level scaling points in a useful direction, but it is not a
complete model.  The remaining error still indicates a coupled multi-contact
allocation problem.

\subsubsection{Vector-Rebound Shadow Diagnostic}

The next theory was that rebound might be returning stored momentum along a
different direction than compression stored it.  To test this without changing
dynamics, each contact now records a compression direction and a current rebound
direction.  The diagnostic also computes a vector-shadow estimate in which
release is returned along the remembered compression direction when that
direction remains compatible with the current normal.

The diagnostic ruled this theory out for the active tests:

\begin{verbatim}
ThreeParticle rebound_dir_dot:
0.99937 .. 1.0

ThreeParticleChase rebound_dir_dot:
1.0 .. 1.0
\end{verbatim}

The vector-shadow estimate did not improve the energy drift:

\begin{verbatim}
ThreeParticle:
actual KE estimate = -0.00024771967622
vector shadow      = -0.00027207492979

ThreeParticleChase:
actual KE estimate = -0.000400429449764
vector shadow      = -0.000400429449764
\end{verbatim}

Therefore the current evidence says the kinetic-energy loss is not primarily
caused by rebound direction drift.  Compression and rebound directions are
nearly identical in the tested cases.

\subsubsection{Elastic Target Compass}

\noindent\textbf{Current analysis commit hash:} \texttt{1deca36ba0f17e82b690b3b80f058a378d2bab5a}

The next diagnostic was not a proposed dynamics rule.  It was a permanent
comparison target.  For each frame, the active closing contact graph is treated
as an elastic hard-contact problem.  The diagnostic solves for contact impulses
that would make the post-contact relative normal velocities satisfy an elastic
response:

\begin{equation}
v_{n,i}^{after} = -v_{n,i}^{before}.
\end{equation}

The solve is coupled over the active contact graph.  Momentum is conserved by
construction because each target contact impulse is equal and opposite across
the unordered pair.  The result is recorded in \texttt{target\_elastic\_*}
fields.

For the batch cases, the elastic target predicted essentially zero net
kinetic-energy drift:

\begin{verbatim}
TwoParticleHorizontal: target KE ~= 0
TwoParticleAngled:     target KE ~= 0
TwoParticleChase:      target KE ~= 0
ThreeParticle:         target KE ~= 0
ThreeParticleChase:    target KE ~= 0
\end{verbatim}

This does not mean the current soft overlap model should directly apply the
target impulse each frame.  The target impulse is the full hard-contact
correction for the instantaneous frame.  The current model is an incremental
compression/rebound model driven by overlap area and stored internal momentum.
The target is therefore a compass: it tells us what an ideal elastic frame
would require, but it is not automatically the Geo response.

\subsubsection{Target-Ratio Shadow Rejection}

The elastic target was then used to test a contact allocation idea.  The
diagnostic kept the current source's total soft applied impulse but redistributed
that total across contacts using the elastic target impulse ratios.

This did not improve the model.  It was especially bad during rebound because
the hard-contact target only exists for closing contacts.  During rebound, the
contacts are separating, so there is no closing target impulse to allocate:

\begin{verbatim}
ThreeParticle:
actual KE      = -0.00024771967622
target ratio   = -0.0022586921383

ThreeParticleChase:
actual KE      = -0.000400429449764
target ratio   = -0.010000167615
\end{verbatim}

The phase split made the failure clear.  In symmetric \texttt{ThreeParticle},
the compression ratio matched the current split, but rebound collapsed because
there was no elastic closing target during release:

\begin{verbatim}
ThreeParticle compression:
actual = -0.00226287573954
ratio  = -0.00226287573954

ThreeParticle release:
actual = +0.00201097246208
ratio  =  0
\end{verbatim}

Therefore target ratios are not a direct rebound allocation rule.

\subsubsection{Vector Internal Momentum Shadow}

The next experiment tested whether the internal momentum ledger should carry
vector history instead of only a scalar amount.  A shadow ledger was added:

\begin{verbatim}
particle.contact_vector_internal_delta[TargetID]
\end{verbatim}

This stores the source compression momentum delta vector accumulated for the
contact.  The first attempted sign convention released the opposite of that
stored vector.  That failed even for two-particle baselines, proving the sign
was wrong.  The corrected convention releases along the stored compression
delta direction and drains the stored vector by subtracting the released
portion.

With the corrected sign, the vector-internal shadow matches the two-particle
baselines:

\begin{verbatim}
TwoParticleHorizontal: vecInternal ~= 0
TwoParticleAngled:     vecInternal ~= 0
TwoParticleChase:      vecInternal ~= 0
\end{verbatim}

However, it does not fix the three-particle cases:

\begin{verbatim}
ThreeParticle:
actual       = -0.00024771967622
sourceShadow = -0.000228391853378
vecInternal  = -0.000339237337509

ThreeParticleChase:
actual       = -0.000400429449764
sourceShadow = -0.000350215634333
vecInternal  = -0.000400429449764
\end{verbatim}

This rules out simple vector internal momentum as the next dynamics rule.  It
is a useful diagnostic, but the best improvement remains the source-scaled
shadow.  The unresolved issue is still coupled multi-contact impulse magnitude
selection, not rebound direction, target-ratio allocation, or a simple vector
ledger replacement.

\subsubsection{Current Interpretation}

The remaining problem is impulse allocation in simultaneous multi-contact
frames.  In \texttt{ThreeParticle}, particle 1 contacts particles 2 and 3 at
the same time.  Particle 1 has a two-contact source context, while particles 2
and 3 each have a one-contact context.  Source-local area weighting gives a
reasonable candidate split, but pair compatibility still reconciles contacts
one unordered pair at a time.

That pair-compatible value is currently conservative.  It preserves total
momentum, but it can still produce the wrong balanced exchange:

\begin{equation}
\Delta \mathbf{p}_1
=
-(\Delta \mathbf{p}_2 + \Delta \mathbf{p}_3),
\qquad
\Delta K_{group} < 0.
\end{equation}

The next dynamics work should therefore focus on the coupled selection of
multi-contact impulse magnitudes.  The useful question is no longer whether the
stored momentum is returned along the right direction, whether hard-contact
target ratios should be copied into rebound, or whether a simple vector ledger
is enough.  The remaining question is whether the simultaneous contact group
chooses the right impulse amounts for each contact while preserving the
source-owned GLSL write rule.

\subsection{Fixed P-Zero Shadow for the Kinetic-Energy Shape}

\subsubsection{Problem}

After the source-owned momentum diagnostics were separated into plot files, the
three-particle case showed a more precise failure mode.  The per-contact
allocation ledgers for particle 1 were internally consistent:

\begin{equation}
I_1 = I_{1\rightarrow2} + I_{1\rightarrow3}.
\end{equation}

However, the particle 1 kinetic-energy curve did not follow the applied
internal momentum through the full collision.  From first contact to near
maximum compression the curves had the same shape, but during release they
spread apart.  This suggested that the remaining problem was not contact
discovery, reporting timing, or rebound direction, but the shape of the
impulse profile used to move between kinetic velocity and stored internal
momentum.

The project state for this investigation began from commit
\texttt{a75a0a6}.

\subsubsection{Investigation}

The old geometry model used a p-zero reference.  It did not make raw momentum
\(m|v|\) parabolic directly.  Instead it computed a zero-velocity overlap from
incoming normal momentum and then mapped current overlap to a progress value:

\begin{equation}
f = \operatorname{clamp}\left(\frac{A}{A_0},0,1\right),
\qquad
I_{target} = P_0 \left(1-\sqrt{1-f}\right).
\end{equation}

A first shadow attempted to recompute \(P_0\) from the current internal
momentum.  That produced a curve that tried to become parabolic but flattened
into a constant, because the reference was being redefined from the value being
measured.

A later review separated two ideas that had been mixed together:

\begin{itemize}[leftmargin=*]
    \item the p-zero reference, which defines the maximum-compression contact
    state;
    \item the target curve used to move the internal momentum ledger toward
    that state.
\end{itemize}

The useful diagnostic question became whether the applied impulse should be the
difference between the current internal momentum and a direct contact-progress
target, rather than a raw area impulse clipped by available momentum.

\subsubsection{Fix}

A diagnostic-only fixed-reference p-zero shadow was added.  For each directed
contact, the shadow stores a contact-owned reference:

\begin{verbatim}
particle.contact_parabolic_zero_reference[TargetID]
\end{verbatim}

The reference is set when the contact first has closing normal momentum:

\begin{equation}
P_{0,k} = \mu_k \max(0,-v_{n,k}).
\end{equation}

The corresponding p-zero overlap area \(A_{0,k}\) is computed from the
force-law relation.  During the active contact, the shadow uses this fixed
reference to compute the progress variable:

\begin{equation}
f_k = \operatorname{clamp}(A_k/A_{0,k},0,1).
\end{equation}

The original fixed-reference shadow used:

\begin{equation}
I^{shadow}_{k}
=
P_{0,k}\left(1-\sqrt{1-\operatorname{clamp}(A_k/A_{0,k},0,1)}\right).
\end{equation}

The current diagnostic shadow has been changed to a direct parabolic target:

\begin{equation}
I^{shadow}_{k}
=
P_{0,k}\left(1-\left(1-f_k\right)^2\right).
\end{equation}

The shadow impulse is then:

\begin{equation}
\Delta I^{shadow}_{k}
=
I^{shadow}_{k}-I^{current}_{k}.
\end{equation}

The shadow records:

\begin{verbatim}
parabolic_zero_overlap_area
parabolic_overlap_fraction
parabolic_zero_internal_mom
parabolic_target_internal_mom
parabolic_delta_impulse
parabolic_compression_impulse
parabolic_release_impulse
parabolic_remaining_internal_mom
\end{verbatim}

These values are written to \texttt{contacts.csv}.  The diagnostics tool also
writes \texttt{MomentumDiagnostics\_top.csv},
\texttt{MomentumDiagnostics\_middle.csv}, and
\texttt{MomentumDiagnostics\_bottom.csv} beside each plot so the plotted values
can be inspected directly.  The bottom plot includes the parabolic target
beside the applied internal momentum and cumulative compression/release
diagnostics.

\subsubsection{Current Result}

With the fixed reference, the p-zero shadow matches the kinetic-energy shape
closely enough to remain the strongest current lead.  The latest diagnostic
uses the direct parabolic target above so the plotted shadow value is explicitly
a contact-progress target.  This is the strongest current evidence that the
next applied dynamics change should move away from frame-local

\begin{verbatim}
compression_impulse = min(raw_impulse, weighted_available_momentum)
release_impulse = min(raw_impulse, release_capacity, stored_internal_momentum)
\end{verbatim}

toward a target-ledger impulse:

\begin{equation}
\Delta I_k = I^{target}_{k} - I^{current}_{k}.
\end{equation}

This has not been promoted to the applied dynamics yet.  The current checkpoint
keeps the p-zero/parabolic result as a shadow diagnostic so the project can be
checked in before the larger model change.

\subsubsection{Backed-Out Promotion Attempt}

After the fixed p-zero shadow looked promising, a first promotion attempt used
the target ledger to choose applied compression/release impulses.  It fixed the
two-particle horizontal p-zero stall only by adding special turnaround and
returning-phase branches.  Those branches made the decision tree too abrupt:
the path depended on separate tests for closing, returning, exact p-zero,
source coupling, and pair compatibility.

That promotion was backed out to commit \texttt{3296158}.  The lesson from the
failed attempt is that p-zero should not be a separate escape rule.  The next
attempt should treat p-zero as part of a single continuous contact-progress
function:

\begin{verbatim}
contact geometry -> contact progress -> target internal momentum
target internal momentum - current internal momentum -> planned impulse
\end{verbatim}

In that form, positive \(\Delta I\) means compression and negative
\(\Delta I\) means rebound.  The applied code should not need a separate
``kick'' at p-zero.

\subsubsection{Standalone Shadow Dynamics Checkpoint}

After the backed-out promotion attempt, the shadow work was separated from the
live dynamics path.  The checkpoint for this cleanup began from commit
\texttt{3dc5296}.

The code now follows the object split:

\begin{verbatim}
GeoBase + GeoDynamics       -> live object
ShadowBase + ShadowDynamics -> shadow object
\end{verbatim}

Both objects are created from the same configuration data, but they own
separate particle lists, contact ledgers, phases, velocities, positions, and
reporting streams.  The shadow path reports to \texttt{*_s.csv}.  The old
in-\texttt{GeoDynamics.py} shadow/parabolic/vector diagnostic fields were
removed from both live and shadow dynamics so the next experiment can be
implemented cleanly in \texttt{ShadowDynamics.py}.

The next step is to apply the parabolic velocity response in the shadow path:

\begin{verbatim}
contact geometry -> contact progress -> target internal momentum
target internal momentum - current internal momentum -> planned impulse
planned impulse -> shadow velocity update
\end{verbatim}

The compression-to-rebound flip should change from a branch-heavy p-zero
escape rule to a velocity-magnitude tolerance:

\begin{equation}
|v_{n,k}| \leq \epsilon_v .
\end{equation}

Once a contact enters the returning phase, that phase should persist until the
shadow contact-owned internal momentum drains or the contact ends.  This keeps
p-zero as part of the continuous response curve rather than as a separate
special-case kick.

\subsubsection{Normal Velocity Target Direction}

The latest shadow tests improved \texttt{TwoParticleHorizontal}: the
contact-owned rebound reference drains internal momentum and restores the
horizontal rebound speed.  However, \texttt{TwoParticleAngled} still loses
kinetic energy after rebound.  The capture files show the loss after the
shadow contact has ended:

\begin{verbatim}
momentum_s.csv:
total_internal_mom = 0
ke_drift           < 0
\end{verbatim}

The particle files show that the loss is symmetric: both particles end with
the same reduced kinetic energy.  This means the current problem is not
unbalanced momentum between particles.  It is that the scalar internal
momentum profile is not producing the intended final velocity magnitude when
the contact normal changes during an angled collision.

This does not change the convention that energy is diagnostic only.  The next
shadow direction should still be velocity/momentum based.  The issue is where
the parabolic profile is imposed.

The current shadow rule imposes the parabola on stored internal momentum:

\begin{verbatim}
progress -> target_internal_momentum
target_internal_momentum - current_internal_momentum -> impulse
\end{verbatim}

The next test should impose the profile on relative normal velocity or normal
momentum:

\begin{verbatim}
progress -> target_relative_normal_velocity
current_normal_momentum -> target_normal_momentum
target_normal_momentum - current_normal_momentum -> impulse
\end{verbatim}

For compression this means reducing the current closing normal momentum toward
a target closing normal momentum.  For rebound this means increasing the
separating normal momentum toward a target separating normal momentum, while
still capping release by the contact-owned internal momentum ledger.  Kinetic
energy remains an output diagnostic, not a dynamics input.

\subsubsection{Normal Velocity Shadow Result and Source Allocation Finding}

At checkpoint \texttt{2252dd1}, the shadow path was updated so the parabolic
profile targets relative normal velocity rather than scalar internal momentum.
The shadow still converts the velocity target to a normal impulse through
reduced mass, so the dynamics input remains momentum based:

\begin{verbatim}
progress -> target_relative_normal_velocity
current_normal_momentum -> target_normal_momentum
target_normal_momentum - current_normal_momentum -> impulse
\end{verbatim}

This produced a strong two-particle result:

\begin{verbatim}
TwoParticleHorizontal: KE returns to start
TwoParticleAngled:     KE returns to start
\end{verbatim}

\texttt{ThreeParticle} still loses kinetic energy even though total momentum
and total internal momentum balance at the end of the reported window:

\begin{verbatim}
start_total_p        = 0.05000000
curr_total_p         = 0.05000000
total_internal_mom   = 0.00000000
KE drift             = -0.00024861
\end{verbatim}

The new diagnostic plots and contact captures show why.  Particle 2 and
particle 3 each have a single dominant contact and their profiles look
correct.  Particle 1 has two symmetric contacts.  At peak storage, particle 2
and particle 3 each reach approximately:

\begin{verbatim}
p2 internal_mom peak = 0.03408681
p3 internal_mom peak = 0.03408681
\end{verbatim}

If particle 1's source compression capacity were shared symmetrically, its
total internal momentum would be expected to peak near:

\begin{verbatim}
0.5 * p2_peak + 0.5 * p3_peak = 0.03408681
\end{verbatim}

Instead, particle 1 peaks higher:

\begin{verbatim}
p1 internal_mom peak = 0.06160446
\end{verbatim}

This means the current shadow is still storing too much scalar internal
momentum for a simultaneous multi-contact source.  In addition, when particle
1 releases to particle 2 and particle 3, the two contact-normal impulses
vector-sum; the transverse components cancel and only the shared source-level
component contributes to particle 1's final velocity.

Therefore the remaining problem is not basic rebound and not a simple missing
scalar ledger.  The next rule must compute a source-level multi-contact
storage/allocation target first, then distribute compatible per-contact
impulses from that source-level result.
