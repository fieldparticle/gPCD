\section{Flow Dynamics Conventions}

\subsection{Current Simple-Model Direction}

Scheduled reservoir birth, deferred admission, and outlet retirement have
been retired from the live Python \texttt{ForceDynamics} path.  Ordinary
mobile particles are generated active from frame zero.  Generator-owned
release-frame records are allowed only for focused bring-up paths such as
\texttt{GenStreaming.py} and \texttt{GenLightingData.py}; they are
precomputed visibility/dynamics schedules, not reservoir birth systems.  The
current verification target is the simple particle model: mobile particles,
boundary-particle locality markers, and stationary parametric walls generated
from \texttt{curve\_wall\_segments}.  Reservoir work is now being reintroduced
as a generator extension of the same simple model rather than as a separate
wall or lifecycle system.  CD-nozzle flow remains separate until the generic
simple and reservoir paths are stable.

The former deferred-birth design remains documented as a historical
investigation in \texttt{ProblemEvolution.tex}.  It is not an active
generation or dynamics rule.

\subsection{FreeStream Streaming Bring-Up}

\texttt{FreeStream} is a focused streaming-inlet bring-up path, not the
replacement reservoir or compressor piston model.  Its generator is:
\begin{verbatim}
gbase/GenStreaming.py
cfg_gendata/FreeStream.cfg
\end{verbatim}

The generator emits mobile particles with timed release frames stored in
\texttt{Data.w}.  This reuses the lifecycle field for a controlled inlet
experiment, but it does not reintroduce the retired reservoir birth helpers
into the active \texttt{ForceDynamics} Python path.  The distinction is:
\begin{itemize}
    \item active \texttt{ForceDynamics} simple and reservoir Python runs use
          active generated mobile particles and permanent dead state;
    \item \texttt{FreeStream} uses generated timed-release records to test
          inlet streaming in the Vulkan simple wrapper family;
    \item \texttt{FreeStream} is not piston driven and must not include the
          optional simple piston GLSL helper.
\end{itemize}

Freestream boundary particles are the same generic function-wall locality
markers used by the simple path.  They use \texttt{ptype=2} and do not
store wall evaluator IDs in \texttt{Data.z}.

\subsection{Lighting Release Generator}

\texttt{GenLightingData.py} inherits the FreeStream release-frame machinery
from \texttt{GenStreaming.py}, but it uses explicit
\texttt{PARTICLE\_DATA} records rather than inlet-wave population.  Each
configured particle may provide:
\begin{verbatim}
releaseFrame
\end{verbatim}
The generator copies this value into \texttt{pdata.state\_flg}, which becomes
\texttt{Data.w} in the runtime particle record.  A missing value defaults to
zero, which means active from frame zero.  Positive values are pending release
frames, and negative values are rejected by the generator.

This generator is for lighting, 3D visualization, and small explicit motion
sequences.  It does not reintroduce reservoir birth admission, does not pack a
reservoir, and does not move particles into an inlet position at runtime.
Material assignment still belongs to the ordinary particle/material contract:
explicit particles carry \texttt{material\_id}, while shared material behavior
comes from \texttt{material\_properties}.

The active generic generator contract is:
\begin{itemize}
    \item mobile particles are active at frame zero;
    \item boundary particles use \texttt{ptype=2};
    \item \texttt{Data.z} is not used for wall evaluator selection;
    \item all stationary wall geometry is in
          \texttt{curve\_wall\_segments};
    \item \texttt{death\_bounds} define permanent mobile-particle death;
    \item \texttt{cell\_array\_width},
          \texttt{cell\_array\_height}, and
          \texttt{cell\_array\_depth} define the rectangular cell domain;
    \item legacy derived \texttt{boundary\_*} test-file fields are not
          emitted by \texttt{GenericGenData}.
\end{itemize}

Death bounds are always part of the simple model contract.  Generated simple
GLSL must read \texttt{death\_x/y/z\_min/max} unconditionally and mark mobile
particles dead before they can continue to the harder cell-array boundary.
There is no \texttt{DEATH\_BOUNDS\_AVAILABLE} feature switch in the current
simple path.

\subsection{Reservoir Generator}

The active replacement work has moved to:
\begin{verbatim}
gbase/GenReservoir.py
cfg_gendata/TestReservoir.cfg
\end{verbatim}

\texttt{GenReservoir.py} inherits the generic particle/wall generator and
adds packed mobile-particle population plus piston metadata.  Its cfg does not
select a wall model, wall function, conduit axis, birth schedule, or
retirement plane.  The reduced Python profile remains selected through
\texttt{sub\_BPRPPy.cfg}; Vulkan uses the same generated data contract.

The packed physical domain is derived from the configured curves rather than
from a second rectangular chamber record.  The reservoir configuration uses
\texttt{curve\_wall\_segments} with generator-only role metadata:
\begin{itemize}
    \item \texttt{curve\_role=1}: packing curve;
    \item \texttt{curve\_role=2}: boundary curve.
\end{itemize}
Packing curves define the upstream chamber used for mobile-particle
population and piston travel.  Boundary curves define stationary collision
walls.  Runtime GLSL does not branch on curve role; role is generation
metadata.

For the horizontal model, positive (x) is axial.  The generator derives the
reservoir packing \(x\) and \(y\) spans from \texttt{curve\_role=1} curves and
validates them against the explicit rectangular cell domain defined by
\texttt{cell\_array\_width}, \texttt{cell\_array\_height}, and
\texttt{cell\_array\_depth}.  The configuration item
\texttt{packing\_z\_bounds} supplies the Z span:
\begin{verbatim}
packing_z_bounds = [z_front, z_back];
\end{verbatim}

The current two-dimensional convention is:
\[
z_{\mathrm{front}}=0,\qquad
z_{\mathrm{back}}=3,\qquad
z_{\mathrm{particle}}=1.
\]
The cell-array depth must therefore be greater than three.

\subsection{Reservoir Packing}

The current generator calculates reservoir capacity and emits the complete
centered mobile-particle grid.  With particle radius (r) and configured
surface-to-surface separation (s), center spacing is:
\[
\ell=2r+s.
\]

Initial particle centers reserve the wall-contact clearance:
\[
c=r(1+o)+\epsilon,
\]
where (o) is \texttt{wall\_contact\_offset}.  The usable center interval
on each packed axis is the physical interval reduced by (c) at both ends.
For usable span (L), the count is:
\[
N=\left\lfloor\frac{L}{\ell}\right\rfloor+1.
\]
The occupied grid is centered in any remaining space so opposing wall margins
are equal.

Generated mobile particles are active from frame zero.  Their initial velocity
comes from \texttt{initial\_particle\_velocity}.  Piston motion uses
\texttt{piston\_velocity}.  During bring-up these may be configured to the
same vector, but they are separate concepts: one initializes mobile
particles, the other moves the analytic piston.

\subsection{Boundary Marker Geometry}

The earlier reservoir prototype used two named finite-line collections:
\begin{verbatim}
linear_chamber_segments = (...);
linear_wall_segments = (...);
\end{verbatim}
Each row is \texttt{[A,B,C,xStart,yStart,xEnd,yEnd,wallFlag]} and represents
the finite line \(Ax+By+C=0\).  The normalized vector \((A,B)\) is the outward
wall normal.  The collections distinguish chamber walls from attached
conduit or nozzle walls without introducing different collision equations.

The simple generic path replaces those line-specific arrays with
\texttt{curve\_wall\_segments}.  Straight walls are represented as cubic
segments with zero higher-order coefficients, so horizontal, vertical,
sloped, and curved walls share one evaluator.  Markers are no farther than one
cell apart and shared coordinates are emitted once per physical wall flag.
Boundary markers remain occupancy placeholders rather than mobile dynamics
sources.

The fixed vertical inlet marker row has been removed.  The analytic piston
occupies that physical role without becoming a boundary particle.  The
generator writes both binary particle data and test metadata.

\subsection{Current Piston Work}

\texttt{piston\_start\_frame} controls when the rear plane begins moving.
The piston starts at the minimum \(x\) extent of the packing curves minus
\texttt{piston\_start\_offset}, advances with
\texttt{piston\_velocity}, and stops at the maximum \(x\) extent of the
packing curves.  The generator emits those derived values as
\texttt{piston\_x\_start} and \texttt{piston\_x\_stop}.  When Python runs
directly from a source cfg, runtime configuration normalization derives the
same bounds from the reservoir curves and infers \texttt{piston\_enabled}
from \texttt{piston\_velocity} when the explicit flag is omitted.  Generated
test files may provide \texttt{piston\_velocity\_x/y/z}; Python normalizes
those scalar values back into \texttt{piston\_velocity}.  Each active source
evaluates the piston once before target scanning.  Python diagnostics report
piston position, velocity, contact count, impulse, and momentum transfer.

The active Python piston is an analytic plane at \texttt{piston\_x}.  It is
not a ghost particle offset behind the displayed line.  Particle-piston
penetration is evaluated from the particle edge and the plane position:
\[
    \delta = r - (x_{particle}-x_{piston}).
\]
This keeps the visual piston line, the moving-wall target velocity, and the
physical contact plane aligned.

The simple GLSL exporter now emits an optional
\texttt{ForceDynamicsSimplePiston.glsl} include for reservoir wrappers.  This
keeps piston support separate from the base simple particle/wall path while
preserving the same source-owned contact schedule.  Reservoir wrappers that
enable the piston must include the piston file and provide
\texttt{piston\_start\_frame}, \texttt{piston\_x\_start},
\texttt{piston\_x\_stop}, and \texttt{piston\_velocity\_x/y/z}.  The simple
piston include does not require or read \texttt{CHAMBER\_MIN/MAX} constants.
