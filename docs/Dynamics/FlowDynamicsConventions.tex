\section{Simple Reservoir Flow}

\subsection{Purpose}

\texttt{SimpleReservoir} is a particle birth/life/death process used with
\texttt{ForceDynamics}.  It does not change the ForceDynamics overlap-area
force law.  It decides when particles become active, which particles are
eligible for dynamics, and when particles leave the active simulation.

\subsection{State Field}

For \texttt{SimpleReservoir}, the model-owned particle state is stored in
\texttt{Particle.Data.w}.  When particles are loaded from a binary
\texttt{pdata} file, the input field \texttt{state\_flg} is copied into
\texttt{Particle.Data.w}.

The meaning of \texttt{Data.w} is:

\begin{itemize}
    \item \(\texttt{Data.w} > 0\): unborn particle.  The value is the frame on
    which the particle becomes active.
    \item \(\texttt{Data.w} = 0\): live particle.  The particle participates in
    ForceDynamics.
    \item \(\texttt{Data.w} < 0\): dead particle.  The particle is removed from
    ForceDynamics.
\end{itemize}

These meanings apply only when the active configuration selects
\texttt{SimpleReservoir}.  Other models may assign different meanings to
\texttt{Data.w}.

\subsection{Birth Rule}

At the beginning of each frame, before contact detection, every unborn
particle is tested against the current frame number:

\[
\texttt{if Data.w > 0 and frameNum >= Data.w}
\]

then the particle is born.  Birth performs:

\begin{itemize}
    \item set \(\texttt{Data.w}=0\);
    \item place the particle at the inlet birth position;
    \item assign the inlet velocity;
    \item make both position buffers agree on the birth position.
\end{itemize}

A particle is born before the frame-start snapshot is built, so the newly born
particle participates in the current frame using ordinary ForceDynamics rules.

In the GLSL compute path this rule is owned by:

\begin{verbatim}
StartReservoir(SourceID)
\end{verbatim}

The compute shader calls this function before particle contact scans, wall
contact scans, velocity update, or position update.  If the function returns
\texttt{false}, the source invocation returns immediately.  This means unborn
and dead particles do not participate in the current frame.  If the source
particle reaches its stored birth frame, \texttt{StartReservoir} resets the
active position to the inlet birth location, updates both position buffers,
sets \texttt{Data.w=0}, and returns \texttt{true}.

\subsection{Live Rule}

A particle is live when:

\[
\texttt{Data.w = 0}.
\]

Only live particles are eligible for:

\begin{itemize}
    \item particle-particle contact detection;
    \item wall contact detection;
    \item force accumulation;
    \item velocity update;
    \item position update;
    \item energy and momentum diagnostics.
\end{itemize}

Unborn and dead particles are skipped by the dynamics pass.

\subsection{Death Rule}

After position update, each live particle is tested against the outlet
boundary.  If the particle has crossed the outlet boundary, then:

\[
\texttt{Data.w = -1}.
\]

A dead particle no longer participates in ForceDynamics and is not reused by
\texttt{SimpleReservoir}.

In the GLSL compute path this rule is owned by:

\begin{verbatim}
RetireParticlePastXMax(SourceID)
\end{verbatim}

The compute shader calls this function after \texttt{CalcPosition(SourceID)}.
It reads the next position buffer, the buffer just written by
\texttt{CalcPosition}, and sets \texttt{Data.w=-1} when the particle has
passed the selected outlet boundary.  For horizontal reservoir flow this is
the \(x_{\max}\) outlet.  For vertical reservoir flow this is the
\(y_{\max}\) outlet.  The legacy function name is retained for shader-wrapper
compatibility even though the implementation now respects the generated
\texttt{PERIODIC\_DIRECTION} selector.

\subsection{Boundary Rule}

For \texttt{SimpleReservoir}, the inlet and outlet are not solid walls.  They
are lifecycle planes.  Only the pipe walls are solid ForceDynamics walls.

For the current two-dimensional pipe convention:

\begin{itemize}
    \item \(x_{\min}\): inlet birth plane, not a wall;
    \item \(x_{\max}\): outlet death plane, not a wall;
    \item \(y_{\min}\): solid lower wall;
    \item \(y_{\max}\): solid upper wall.
\end{itemize}

Therefore the wall scan for \texttt{SimpleReservoir} includes only wall flags
3 and 4.

\subsection{Boundary Particle Wall Rule}

Reservoir configurations may select the wall representation with:

\begin{verbatim}
wall_type = 0;  // manual configured wall planes
wall_type = 1;  // particle boundary markers
\end{verbatim}

When \texttt{wall\_type=0}, the configured wall fields retain their legacy
meaning as physical wall planes.  When \texttt{wall\_type=1}, boundary
particles define the solid wall broad phase.  The configured manual wall
planes must not be used as the physical wall-collision geometry for those
pipe walls.

A boundary particle is an occupancy marker, not a moving dynamics particle.
It is stored in the same particle buffer with \texttt{ptype=1.0} and
\texttt{Data.w=0.0} so the vertex/cell placement pass includes it.  The
compute/source dynamics dispatch still processes only mobile particles as
sources.  When a mobile source encounters a boundary marker as a target, it
must evaluate the wall segment represented by that marker rather than process
the marker as an ordinary particle-particle collision.

The current straight-wall implementation is:

\begin{verbatim}
EvaluateWallSegment(SourceID, BoundaryID)
\end{verbatim}

For now, each boundary marker represents either a straight horizontal or
straight vertical wall segment through the center of the marker's cell.  The
selected wall function is generated into \texttt{boundary.glsl} as
\texttt{WALL\_FUNC}.  Horizontal segments classify markers as lower or upper
walls.  Vertical segments classify markers as left or right walls.  The
evaluator calculates the current source-to-segment wall ghost geometry and
returns the normal, overlap area, center distance, and wall flag needed by the
ordinary wall contact slot.  Later sloped or curved boundaries should extend
\texttt{EvaluateWallSegment} without changing the source-owned force
accumulation rule.

\subsection{ForceDynamics Invariance}

\texttt{SimpleReservoir} does not modify the ForceDynamics equations:

\[
F_{ij}=q_{ij}A_{ij}
\]

and

\[
\mathbf v_{n+1}
=
\mathbf v_n+\frac{\mathbf F_n}{m}\Delta t.
\]

It only controls particle eligibility and boundary interpretation.


\subsection{Unborn Particle Rule}

Reservoir particles may be stored in repeated columns of length
\texttt{col\_length}.  Many unborn particles may therefore share the same
stored position.  These stored positions are not physical particle positions
until the particle is born.

For \texttt{SimpleReservoir}, if:

\[
\texttt{Data.w > frameNum}
\]

then the particle is unborn.  An unborn particle is ignored by all
ForceDynamics operations:

\begin{itemize}
    \item no particle-particle contact check;
    \item no wall contact check;
    \item no tunneling check;
    \item no starting-contact check;
    \item no force accumulation;
    \item no velocity update;
    \item no position update;
    \item no energy or momentum diagnostics.
\end{itemize}

This prevents staged reservoir particles that share the same stored location
from producing artificial overlap, tunneling, or starting-collision errors.

\subsection{Birth Frame Rule}

When:

\[
\texttt{Data.w <= frameNum}
\]

and \(\texttt{Data.w > 0}\), the particle becomes live.  The birth process
sets:

\[
\texttt{Data.w = 0}
\]

and assigns the particle's active inlet position and velocity before the
ForceDynamics frame snapshot is built.

The corresponding GLSL source invocation order is:

\begin{verbatim}
if (!StartReservoir(SourceID))
    return;

// particle contact scan
// wall contact scan
CalcVelocity(SourceID, totalForce);
CalcPosition(SourceID);
RetireParticlePastXMax(SourceID);
\end{verbatim}

This preserves source ownership.  The reservoir state gates only the current
source invocation; it does not write target particles and does not introduce a
separate global reservoir pass.

The generated \texttt{PERIODIC\_DIRECTION} selector controls whether the inlet
and outlet are interpreted along \(x\) or \(y\).  When
\texttt{PERIODIC\_DIRECTION} is \texttt{vertical}, birth preserves the
generated \(x\) coordinate and offsets the \(y\) coordinate from
\(y_{\min}\).  Retirement then checks \(y_{\max}\).  When the selector is not
vertical, the historical horizontal \(x\)-axis behavior is used.


\subsection{Birth Placement Rule}

When an unborn particle becomes live, its stored reservoir position is not used
as its active physical position.  The particle is moved to the inlet birth
position before ForceDynamics contact detection.

The birth position is offset from the reservoir/staging column by at least

\[
1.1(2r)
\]

where \(r\) is the particle radius.  Equivalently, the minimum birth offset is

\[
2.2r.
\]

This places the newly live particle slightly more than one particle diameter
away from particles that remain in the reservoir column.  The factor \(1.1\)
adds a small clearance margin so the particle is not born in contact due to
rounding or exact-diameter placement.

After birth:

\[
\texttt{Data.w}=0
\]

and both position buffers are set to the active birth position.

\subsection{Death Rule}

After position update, every live particle is tested against the outlet
boundary using its center position.  If the particle center has passed the
outlet boundary, then:

\[
\texttt{Data.w}=-1.
\]

A particle with \(\texttt{Data.w}<0\) is dead.  Dead particles are no longer
processed by ForceDynamics and are not checked for collision, wall contact,
tunneling, starting overlap, force, motion, or ordinary energy/momentum
diagnostics.

\subsection{Vertex Lifecycle Rule}

The graphics vertex stage must apply the same lifecycle rule before building
particle cell occupancy or drawing particle markers.  A vertex invocation for
an unborn or dead reservoir particle must not write cell-array occupancy and
must not leave vertex outputs undefined.

The shared GLSL vertex helper is:

\begin{verbatim}
ParticleLifeActive(SourceID, frameNum)
\end{verbatim}

For inactive particles, the vertex shader must write a defined offscreen
position before returning:

\begin{verbatim}
if (!ParticleLifeActive(index, uint(ShaderFlags.frameNum))) {
    gl_Position = vec4(2.0, 2.0, 2.0, 1.0);
    gl_PointSize = 0.0;
    fragColor = vec3(0.0);
    return;
}
\end{verbatim}

A bare vertex-stage \texttt{return} is not sufficient because it prevents
occupancy processing but can leave \texttt{gl\_Position} and color outputs
undefined.  \texttt{discard} is not available in vertex shaders, so moving the
vertex outside clip space is the required suppression method.
