\section{Flow Dynamics Conventions}

\subsection{Current Reservoir Direction}

Scheduled particle birth, lifecycle gating through \texttt{Data.w}, and
outlet retirement have been retired from the live Python
\texttt{ForceDynamics} path.  Mobile particles are generated active from
frame zero.  The selected replacement is a finite, prefilled reservoir whose
rear wall will later move as a piston.  The piston model is not yet
implemented.

The former deferred-birth design remains documented as a historical
investigation in \texttt{DynamicsConventions.tex} and
\texttt{ProblemEvolution.tex}.  It is not an active generation or dynamics
rule.

\subsection{Horizontal Reservoir Generator}

The active replacement work begins in:
\begin{verbatim}
gbase/BoundaryParticleReservoirHorizontal.py
cfg_gendata/BoundaryParticleReservoirHorizontal.cfg
\end{verbatim}

The generator owns the horizontal orientation.  Its cfg does not select a
wall model, wall function, conduit axis, birth schedule, or retirement plane.
The reduced Python profile remains selected through
\texttt{sub\_BPRPPy.cfg}; the Vulkan profile is deferred until Python
geometry and dynamics are verified.

The physical domains are configured as:
\begin{verbatim}
reservoir_bounds = [
    x_start, x_end,
    y_bottom, y_top,
    z_front, z_back
];

pipe_bounds = [
    x_start, x_end,
    y_bottom, y_top,
    z_front, z_back
];
\end{verbatim}

For the horizontal model, positive (x) is axial.  The reservoir outlet and
pipe inlet must share a plane:
\[
x_{r,\max}=x_{p,\min}.
\]
The pipe (y) and (z) bounds must fit inside the reservoir outlet face.
Every bound must lie inside the explicit rectangular cell domain defined by
\texttt{cell\_array\_width}, \texttt{cell\_array\_height}, and
\texttt{cell\_array\_depth}.

The current two-dimensional convention is:
\[
z_{\mathrm{front}}=0,\qquad
z_{\mathrm{back}}=3,\qquad
z_{\mathrm{particle}}=1.
\]
The cell-array depth must therefore be greater than three.

\subsection{Reservoir Packing Report}

The current generator calculates reservoir capacity but does not yet create
mobile particle records.  With particle radius (r) and configured
surface-to-surface separation (s), center spacing is:
\[
\ell=2r+s.
\]

Initial particle centers reserve the wall-contact clearance:
\[
c=r(1+o)+\epsilon,
\]
where (o) is \texttt{wall\_contact\_offset}.  The usable center interval
on each packed axis is the physical interval reduced by (c) at both ends.
For usable span (L), the count is:
\[
N=\left\lfloor\frac{L}{\ell}\right\rfloor+1.
\]
The occupied grid is centered in any remaining space so opposing wall margins
are equal.

For the current reduced Python case, the report is a \(30\times13\) grid,
or 390 possible mobile particles.  This is capacity only; no mobile particles
are written yet.

\subsection{Boundary Marker Geometry}

The current horizontal generator writes boundary markers only.  It creates:

\begin{itemize}
    \item a closed vertical reservoir inlet;
    \item upper and lower reservoir walls;
    \item upper and lower pipe walls;
    \item vertical shoulder segments where a narrower pipe joins the
    reservoir;
    \item no vertical wall across the pipe outlet.
\end{itemize}

Markers are no farther than one cell apart and shared coordinates are emitted
once.  Horizontal segment markers use evaluator ID 1 and vertical segment
markers use evaluator ID 2 through \texttt{pdata.ptype}; the Vulkan loader
later transports that value to \texttt{Data.z}.  Boundary markers remain
occupancy placeholders rather than mobile dynamics sources.

The current binary-only milestone writes one null record followed by 43
boundary-marker records for the reduced horizontal geometry.  It deliberately
does not write a \texttt{.tst} file.

\subsection{Future Piston Work}

After wall geometry and prefilled particle packing are verified in Python, the
next model stages are:

\begin{enumerate}
    \item materialize the reported nonoverlapping mobile grid;
    \item assign the common initial axial velocity;
    \item define the rear-wall piston start frame;
    \item move the piston at the selected axial speed;
    \item include piston impulse in momentum diagnostics;
    \item only then update generated GLSL and Vulkan wrappers.
\end{enumerate}
