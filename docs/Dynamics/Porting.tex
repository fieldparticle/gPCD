\section{Porting Python to GLSL}

Python is the readable canonical source for GLSL dynamics.  Naming,
structures, and the dynamics schedule should match as closely as practical.
The Python harness may replace GPU cell traversal with a direct scan, but it
must preserve the same source-owned processing order.

The current ForceDynamics compute flow is:
\begin{verbatim}
main()
{
    vec3 totalForce = vec3(0.0);
    uint SourceID = gl_GlobalInvocationID.x;

    InitializeStartingContactState();

    for (uint ii = 0; ii < 8; ii++)
    {
        uint loc = P[SourceID].CornerList[ii].ploc;

        for (uint jj = 0; jj < MAX_CELL_OCCUPANCY; jj++)
        {
            uint TargetID = clink[loc].idx[jj];
            if (TargetID == 0)
                break;

            if (isParticleContact(
                    ii, SourceID, TargetID,
                    uint(ShaderFlags.positionBuffer)) > 0
                && isDuplicate(SourceID) == False)
            {
                ProcessParticleCollision(
                    TargetID, SourceID, totalForce);
            }
        }
    }

    for (uint wall = 1; wall <= 4; wall++)
    {
        if (isWallContact(
                SourceID, wall,
                uint(ShaderFlags.positionBuffer)) > 0)
        {
            ProcessWallCollision(SourceID, wall, totalForce);
        }
    }

    CalcVelocity(
        SourceID, totalForce,
        uint(ShaderFlags.positionBuffer));
    CalcPosition(SourceID, uint(ShaderFlags.positionBuffer));
}
\end{verbatim}

\texttt{totalForce} is a source-local variable passed as an \texttt{inout}
parameter to particle and wall collision processors.  Each processor adds its
force contribution immediately.  Contact slots may be recorded for
diagnostics, but ForceDynamics does not loop over them later to calculate
force.

Live ForceDynamics also carries starting-contact state.  A GLSL port must
preserve that state outside the per-frame contact slots so an initial
overlap can use effective contact geometry until it rebounds to the starting
separation and then clears the physical overlap.

The GPU launches one invocation for each valid source particle, so the
dynamics invocation does not need a source-ID validity check.  The current
wall implementation directly scans four configured walls.  A later boundary
representation may place uniquely identified boundary segments in cells
without changing the source-owned force and integration schedule.
