\section{ForceDynamics Source Flow}

\newcommand{\ForceFunctionSection}[1]{%
  \subsection{ForceDynamics:#1}%
  \label{force:#1}%
}

\texttt{ForceDynamics} treats particle overlap area as a central repulsive
force field.  Each source reads immutable frame-start geometry, calculates the
force contribution from every overlapping target, vector-sums those
contributions, and writes only its own velocity.

\ForceFunctionSection{CollisionRun}

\texttt{CollisionRun} retains the shared frame chain.  Contact forces are
calculated from one immutable frame-start snapshot, velocities are updated,
and movement then uses the updated velocities.  This is a semi-implicit Euler
step.

\ForceFunctionSection{ContactDynamics}

\texttt{ContactDynamics(SourceID)} is the complete source-level force chain:

\begin{verbatim}
BeginSourceContactParms
for each TargetID:
    InitializeContactState
    CalculateContactParms
CalculateContactForces
ApplySourceVelocityDelta
\end{verbatim}

There is no contact weighting, compression/release decision, available
momentum cap, collision phase, or internal-momentum ledger.

\subsection{No Contact History}

Force response is instantaneous and memoryless.  It reads only current
frame-start geometry and fixed particle properties.  Previous overlap area,
contact normal, relative velocity, force, and potential energy are not retained
as dynamics inputs.

\ForceFunctionSection{BeginSourceContactParms}

\texttt{BeginSourceContactParms(SourceID)} resets the source overlap-area total
used for geometry diagnostics.

\ForceFunctionSection{InitializeContactState}

\texttt{InitializeContactState(SourceID, TargetID)} allocates one current-frame
contact slot and populates its source-to-target normal, overlap area, center
distance, and penetration depth.

\ForceFunctionSection{GetContactState}

\texttt{GetContactState(SourceID, TargetID)} obtains a reusable source-owned
contact slot and fills it with current frame-start geometry.  The slot contains
no persistent collision phase or stored momentum.

\ForceFunctionSection{AppendContactSlot}

\texttt{AppendContactSlot(SourceID, TargetID)} records the target ID, particle
contact type, and active-this-frame flag in the next available source-owned
slot.

\ForceFunctionSection{GetContactSlots}

\texttt{GetContactSlots(SourceID)} returns the fixed contact-slot array owned
by the source particle.

\ForceFunctionSection{GetParticleGeometry}

\texttt{GetParticleGeometry(SourceID, TargetID)} calculates geometry from the
immutable frame-start position snapshot.  The returned normal points from
source to target.

\ForceFunctionSection{GetParticlePosition}

\texttt{GetParticlePosition(ParticleID)} returns the immutable frame-start
position when available, ensuring both sides of a pair calculate from the same
geometry.

\ForceFunctionSection{CalculateContactParms}

\texttt{CalculateContactParms} records relative normal velocity for
diagnostics and accumulates overlap area.  Relative velocity does not gate or
cap the force.

\ForceFunctionSection{GetStartFrameVelocity}

\texttt{GetStartFrameVelocity(ParticleID)} returns the immutable frame-start
velocity used only for relative-normal-velocity diagnostics.

\ForceFunctionSection{CalculateContactForces}

For each overlapping pair:

\begin{equation}
F_{ij}=q_{ij}A_{ij}
\end{equation}

\begin{equation}
J_{ij}=F_{ij}\Delta t.
\end{equation}

The force is applied during both approach and separation.  The function also
records pair potential energy:

\begin{equation}
U_{ij}(d)=q_{ij}\int_d^{r_i+r_j}A_{ij}(s)\,ds.
\end{equation}

\ForceFunctionSection{GetPairStiffness}

\texttt{GetPairStiffness(SourceID, TargetID)} returns the nonnegative mean of
the two particle-owned stiffness values:

\begin{equation}
q_{ij}=\max\left(0,\frac{q_i+q_j}{2}\right).
\end{equation}

\ForceFunctionSection{GetContactPotentialEnergy}

\texttt{GetContactPotentialEnergy} evaluates the overlap-area integral using
Simpson integration.  This value is diagnostic; force application depends
directly on the instantaneous overlap area.  Removing this diagnostic function
would not change particle motion.

\ForceFunctionSection{TotalPotentialEnergy}

\texttt{TotalPotentialEnergy} is called by reporting outside the source-owned
contact-response pass.  It sums each unordered particle pair once to avoid
diagnostic double counting.  System energy diagnostics report:

\begin{equation}
E_{\mathrm{total}}=KE+\sum_{i<j}U_{ij}.
\end{equation}

\ForceFunctionSection{ApplySourceVelocityDelta}

Each source contact contributes:

\begin{equation}
\Delta\mathbf p_{ij}=-J_{ij}\mathbf n_{ij}.
\end{equation}

All source contact contributions are vector-summed before one source velocity
write.  Independent opposing-source evaluation produces the equal-and-opposite
target contribution because pair stiffness, overlap area, and frame-start
geometry are symmetric.

\ForceFunctionSection{GetParticleMass}

\texttt{GetParticleMass(ParticleID)} returns particle mass from
\texttt{parms.x} with an epsilon lower bound.

\ForceFunctionSection{Move}

\texttt{Move(SourceID)} advances position using the newly updated velocity:

\begin{equation}
\mathbf x_{n+1}=\mathbf x_n+\mathbf v_{n+1}\Delta t.
\end{equation}

Completed-collision energy error converges toward zero under timestep
refinement.  Potential energy may exchange with kinetic energy while overlap
is active.  At \(\Delta t=0.05\), the current \texttt{ThreeParticle} result is:

\begin{verbatim}
vector momentum drift = 0
final energy difference = approximately -0.00000017
\end{verbatim}

The remaining difference is a small numerical energy leak from the current
integration step.
